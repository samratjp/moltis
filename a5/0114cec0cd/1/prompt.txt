### Risks and Open Questions

1. **Pagination strategy** — Offset/limit is simplest but cursor-based (`updated_at` + `id`) scales better. Other stores in the project don't appear to use pagination yet, so offset/limit is likely sufficient for now.
2. **Search implementation** — SQLite FTS5 exists in the memory crate for chunks. For CRM, simple `LIKE '%query%'` on name/email/phone may suffice initially vs. a full FTS5 table. Email/phone are encrypted with `Secret<String>` — search requires exposing them for the SQL LIKE, which is fine since they're stored as plaintext in SQLite (Secret is a Rust-side guard, not storage encryption).
3. **API surface** — The issue doesn't specify REST vs RPC. The project convention is each tab gets its own namespace (`/api/crm/*` + `crm.*` RPC). This should be a separate issue/phase if not in scope.
4. **No metadata column on matters** — The init migration had `metadata TEXT` but the enrichment migration and current `Matter` struct don't include it. This is fine — the types from P1.2 are authoritative.

### Proposed Approach

1. **Add pagination types** — `PaginationParams` (offset, limit) and `PaginatedResult<T>` in `types.rs`
2. **Extend `CrmStore` trait** — Add `list_contacts_filtered(stage, practice_area, pagination)`, `search_contacts(query)`, `get_contact_with_channels(id)`
3. **Implement in both stores** — SQLite with WHERE clauses + LIMIT/OFFSET, memory with iterator filters
4. **Add `tracing` feature** — Gate `#[instrument]` on store methods
5. **Wire into gateway** — Add `CrmStore` to `GatewayServices`, instantiate `SqliteCrmStore` in server startup
