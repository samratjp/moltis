## STEP 0: NUCLEAR SCOPE CHALLENGE

### 0A. Premise Challenge

**Is this the right problem?** Yes. The CRM has contacts with lifecycle stages (Lead → Prospect → Active → Inactive → Closed) and matters with phases (Intake → Discovery → Negotiation → Resolution → Review → Closed). Currently there's no automated mechanism to check on contacts/matters that need attention. Follow-ups are entirely manual.

**What's the actual user outcome?** A law firm user with contacts in various stages wants the system to periodically check: "Which contacts haven't been contacted in N days?" and "Which matters are stalled in a phase?" and then either notify the user or trigger an agent to draft/send follow-up messages.

**What if we did nothing?** Contacts go cold. Matters stall without anyone noticing. The CRM becomes a write-only database. This is a real pain point for any practice management tool.

---

### DECISION 1: How does the follow-up agent access CRM data?

**Context:** The follow-up cron job fires an `AgentTurn` which runs an isolated agent session. That agent needs to know which contacts/matters are stale. Two approaches:

**A) Prompt enrichment (like heartbeat events enrichment)** — At job fire time, query CRM for stale contacts/matters and inject the results into the prompt text. The agent receives pre-digested data and composes follow-up messages.
- Effort: M | Risk: Low | Simple, deterministic
- Matches heartbeat pattern where system events are baked into prompt

**B) Agent has CRM tools** — The agent prompt instructs it to use CRM query tools to find stale contacts itself. More flexible but requires the isolated session to have CRM tool access.
- Effort: L | Risk: Medium | More powerful but more complex wiring
- The agent could misuse queries, hallucinate contact data

**RECOMMENDATION: Choose A** because it's the minimal-diff approach that mirrors the proven heartbeat enrichment pattern, keeps the agent stateless, and avoids giving an autonomous agent write access to CRM.

---

### DECISION 2: What defines "needs follow-up"?

**Context:** We need heuristics for which contacts/matters get flagged.

**A) Time-since-last-interaction only** — Config field `stale_days: u64` (default 14). Query: contacts where most recent interaction created_at < now - stale_days.
- Effort: S | Simple, covers 80% of cases

**B) Stage-aware rules** — Different thresholds per stage: Leads get 7 days, Prospects 14 days, Active contacts 30 days. Matters in Intake phase get shorter windows than Discovery.
- Effort: M | More sophisticated, better UX

**C) Configurable per-stage thresholds** — `follow_up.stale_days.lead = 7`, `follow_up.stale_days.prospect = 14`, etc.
- Effort: M | Most flexible, but more config surface

**RECOMMENDATION: Choose A** for this PR, with a note that B/C can be added later. A single `stale_days` field is the minimum viable heuristic. The CRM store already has `list_interactions_by_contact()` for the query. Stage-aware rules are a natural Phase 2 enhancement.

---

### DECISION 3: What does the follow-up agent DO with stale contacts?

**A) Notify only** — Agent composes a summary of stale contacts/matters and delivers it to a configured channel (e.g., Slack DM to the user). User then manually follows up.
- Effort: S | Risk: Low | Safe, no autonomous outreach

**B) Draft follow-up messages** — Agent drafts messages but doesn't send them. Stores drafts as CRM interactions of kind `Note`.
- Effort: M | Risk: Low | Useful but requires CRM write access from agent

**C) Send follow-up messages** — Agent composes and sends follow-up messages directly to contacts via their channels.
- Effort: L | Risk: HIGH | Autonomous outreach to real people. Legal/compliance concerns.

**RECOMMENDATION: Choose A** for this initial implementation. Autonomous outreach (C) is a fundamentally different product decision with legal implications. Start with notification, let users manually follow up.
