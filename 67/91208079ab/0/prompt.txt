# Global Agent Instructions

   If blocked, post the blocker details as a Linear comment and stop.
4. Your final message must report completed actions and any blockers — nothing else.

- Spend extra effort on planning and verification.

- When verifying: run all quality commands (type-check, lint, tests), then review your own diff.
- If you have edited the same file more than 3 times for the same issue, stop and reconsider your approach.

1. Run the project's type-check command to verify the codebase compiles clean.
2. Run the project's test command to verify all tests pass.
3. If either fails, investigate and fix before starting new work.

The issue requests four SQLite tables with indexes and a `run_migrations()` function in `lib.rs`:

3. **`crm_interactions`** — logged interactions (messages, calls, meetings)

- `run_migrations()` in `lib.rs` using `sqlx::migrate!("./migrations").set_ignore_missing(true)`
- `SqliteCrmStore` with full CRUD operations against the `contacts` table
- `CrmStore` trait, `MemoryCrmStore`, error types, `Contact` type
- Feature-gated integration in `server.rs` (line 1872-1876) and `cli/Cargo.toml`
- 7 passing tests in `store_sqlite.rs`

### Affected Files

| File | Change |
|------|--------|
| `crates/crm/migrations/20260317000000_init.sql` | **Rename** `contacts` → `crm_contacts`; **add** `crm_matters`, `crm_interactions`, `crm_contact_channels` tables + indexes |
| `crates/crm/src/store_sqlite.rs` | Update all SQL queries from `contacts` → `crm_contacts` |
| `crates/crm/src/types.rs` | Add `Matter`, `Interaction`, `ContactChannel` structs |
| `crates/crm/src/store.rs` | Extend `CrmStore` trait with methods for matters, interactions, contact_channels |

| `crates/cli/src/db_commands.rs` | **Gap:** Add `crm_contacts` (and new tables) to `clear_databases()` table list; add vault+crm migrations to `run_migrations()` |
| `CLAUDE.md` | Update crate-to-table mapping to include CRM tables |

### Risks & Open Questions

1. **Table naming**: The scaffold uses `contacts` but the issue specifies `crm_contacts`. Using the `crm_` prefix avoids namespace collisions with other crates sharing the same `moltis.db`. **Recommendation:** rename to `crm_contacts` as specified.

2. **Migration strategy**: Since P1.1 just landed and there are likely no production databases yet, we can safely modify the existing `20260317000000_init.sql` rather than adding an `ALTER TABLE RENAME` migration. If any dev databases exist, they can `moltis db reset`.

3. **CLI db_commands gap**: The `run_migrations()` in `db_commands.rs` (lines 147-178) is missing vault and CRM migrations. The `clear_databases()` function (lines 71-88) is also missing vault and CRM tables. Both need updating.

4. **Foreign keys**: `crm_interactions` and `crm_contact_channels` will reference `crm_contacts`. `crm_interactions` may also reference `crm_matters`. Need to ensure migration order handles this (all in one file is fine since it's a single crate).

5. **Schema design for `crm_matters`**: The issue doesn't specify fields. Likely: `id`, `title`, `status`, `contact_id` (FK), `metadata`, `created_at`, `updated_at`. Need to decide on `status` enum values.

6. **Schema design for `crm_contact_channels`**: Junction table linking contacts to channels. Fields: `contact_id`, `channel_type`, `channel_identifier`, `verified`, timestamps. Composite PK or separate `id`?

### Proposed Approach

1. Redesign the init migration SQL to create all four `crm_*` tables with proper FKs and indexes in a single migration file
2. Update `Contact` type and `SqliteCrmStore` queries to use `crm_contacts` table name
3. Add new domain types (`Matter`, `Interaction`, `ContactChannel`) in `types.rs`
4. Extend `CrmStore` trait with CRUD for all four entities
5. Implement `SqliteCrmStore` and `MemoryCrmStore` for new entities
6. Fix `db_commands.rs` to include vault and CRM in both `run_migrations()` and `clear_databases()`
7. Add comprehensive tests for all new store operations
8. Update CLAUDE.md crate-to-table mapping
