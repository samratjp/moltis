### Architecture Analysis

**Existing CRM crate** (`crates/crm/`):
- **Types** (`types.rs`): `Contact`, `ContactChannel`, `Matter`, `Interaction` — fully defined with serde, PII-safe `Secret<String>` fields, and comprehensive enums (`ContactStage`, `MatterStatus`, `MatterPhase`, `PracticeArea`, `InteractionKind`).
- **Store trait** (`store.rs`): `CrmStore` — a low-level persistence trait with `SqliteCrmStore` and `MemoryCrmStore` impls. Uses `crate::Result<T>` (CRM-local error type).
- **Error** (`error.rs`): CRM-specific `Error` enum with `Sqlx`, `NotFound`, `Parse`, `External` variants.

**Service-traits crate** (`crates/service-traits/src/lib.rs`):
- Pattern: each domain gets a `#[async_trait] trait XxxService: Send + Sync` with methods returning `ServiceResult<T = Value>` (i.e., `Result<serde_json::Value, ServiceError>`).
- Each trait has a `NoopXxxService` struct with stub impl (list ops return `json!([])`, reads return `json!({})`, writes return `Err("xxx not configured")`).
- All traits aggregated in `pub struct Services` with `Arc<dyn XxxService>` fields and wired in `Default` impl.

**Gateway integration** (`crates/gateway/`):
- `moltis-crm` is an optional dependency behind feature flag `crm`.
- Currently only runs migrations — no API routes or RPC handlers exist for CRM yet.

### Design Decision: Typed vs Value-based API

The existing service traits use `serde_json::Value` for params and returns (`ServiceResult<Value>`). This is the untyped RPC-dispatch pattern used by all other services.

However, the CRM types are well-defined concrete structs. Two approaches:

1. **Match existing pattern** — Use `ServiceResult` (Value-based) like all other traits. The gateway RPC dispatcher already expects this. Simpler, consistent.
2. **Use concrete types** — Return `ServiceResult<Vec<Contact>>` etc. More type-safe but breaks the current dispatch pattern and would require the service-traits crate to depend on moltis-crm types.

**Recommendation**: Use the existing Value-based pattern for consistency. The concrete types live in `moltis-crm` and the gateway impl will serialize/deserialize at the boundary. This keeps `service-traits` dependency-free from domain crates.
