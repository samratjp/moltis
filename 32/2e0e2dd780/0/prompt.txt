> ## Investigation

### Requirements

The issue requests defining these types in `crates/crm/src/types.rs`:

| Type | Kind | Purpose |
|------|------|---------|
| `Contact` | struct | **Already exists** — needs enrichment with PII-protected fields (email, phone) and stage |
| `Matter` | struct | Legal matter / case — links contacts to work items |
| `Interaction` | struct | Logged touchpoint between system and contact |
| `ContactChannel` | struct/enum | Multi-channel identity records per contact |
| `ContactStage` | enum | Lifecycle stage (Lead, Active, Inactive, etc.) |
| `MatterStatus` | enum | Matter lifecycle (Open, Closed, OnHold, etc.) |
| `MatterPhase` | enum | Phase within a matter (Intake, Discovery, Resolution, etc.) |
| `PracticeArea` | enum | Domain categorization for matters |

### Current State

- **`Contact` struct exists** with: id, name, source, external_id, metadata, timestamps
- **`CrmStore` trait exists** with CRUD + external-id lookup
- **SQLite + in-memory stores** implemented with 18 tests
- **Migration exists** for `contacts` table only
- `secrecy` crate is a **workspace dependency** (v0.8 with serde feature) — used extensively in oauth, auth, providers

### Affected Files

| File | Changes |
|------|---------|
| `crates/crm/src/types.rs` | Enrich `Contact`, add `Matter`, `Interaction`, `ContactChannel`, enums |
| `crates/crm/Cargo.toml` | Add `secrecy = { workspace = true }` dependency |
| `crates/crm/src/store.rs` | Extend `CrmStore` trait with matter/interaction CRUD |
| `crates/crm/src/store_sqlite.rs` | New SQL queries + row types for new tables |
| `crates/crm/src/store_memory.rs` | In-memory impls for new store methods |
| `crates/crm/src/error.rs` | Possibly add new error variants |
| `crates/crm/migrations/` | New migration(s) for matters, interactions, contact_channels tables |

### Design Decisions

1. **PII fields on Contact**: `email` and `phone` should use `Secret<String>` per issue requirement and project convention (see `crates/oauth/src/types.rs` for serde helpers pattern)
2. **`source` field → `ContactChannel`**: The existing `source: Option<String>` + `external_id: Option<String>` on Contact should be **kept for backward compat** as the primary channel, but `ContactChannel` enables a contact to have multiple channel identities (one contact across Telegram + WhatsApp + email)
3. **Enums should derive**: `Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize` and use `#[serde(rename_all = "snake_case")]` — matching `ChannelType` pattern in `crates/channels/src/plugin.rs`
4. **Match on types, not strings**: All enums get `as_str()`, `Display`, and `FromStr` impls — no string matching in business logic
5. **`Matter` links to contacts via `contact_id`**: Foreign key relationship, not embedded
6. **`Interaction` links to both**: `contact_id` + optional `matter_id`, with `InteractionKind` enum (Call, Email, Message, Meeting, Note)
7. **Timestamps**: Use `u64` millis consistent with existing `Contact` pattern

### Risks and Open Questions

1. **Serde round-trip for `Secret<String>`**: Need custom serde helpers for DB storage — existing pattern in `crates/oauth/src/types.rs` can be reused, but the CRM may need its own since PII fields are stored in SQLite (not just JSON config files)
2. **Contact backward compatibility**: Existing `Contact::new()` and `Contact::with_source()` constructors must remain functional — new fields should have sensible defaults
3. **Store trait splitting**: Should `CrmStore` grow to include matter/interaction methods, or should there be separate `MatterStore` / `InteractionStore` traits? Single trait is simpler but violates ISP as it grows
4. **Migration ordering**: New migration must come after the existing `20260317000000_init.sql` — suggested name: `20260317000001_crm_types.sql`
5. **`PracticeArea` extensibility**: Should this be a closed enum or support custom values? Closed enum is safer (match on types) but less flexible for non-legal CRM use cases

### Proposed Approach

1. Add `secrecy` dependency to `crates/crm/Cargo.toml`
2. Define enums first: `ContactStage`, `MatterStatus`, `MatterPhase`, `PracticeArea` with full derive set + Display/FromStr
3. Enrich `Contact`: add `email: Option<Secret<String>>`, `phone: Option<Secret<String>>`, `stage: ContactStage` with `Default` impl
4. Define `ContactChannel` struct: `id, contact_id, channel_type, channel_id, display_name, verified, timestamps`
5. Define `Matter` struct: `id, title, contact_id, status, phase, practice_area, description, timestamps`
6. Define `Interaction` struct: `id, contact_id, matter_id (optional), kind, summary, channel, timestamps`
7. Write comprehensive unit tests for all new types (construction, serde round-trip, enum conversions)
