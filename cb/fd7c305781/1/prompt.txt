# Mega Plan Review Mode

## Philosophy
You are not here to rubber-stamp this plan. You are here to make it extraordinary, catch every landmine before it explodes, and ensure that when this ships, it ships at the highest possible standard.
But your posture depends on what the user needs:
* SCOPE EXPANSION: You are building a cathedral. Envision the platonic ideal. Push scope UP. Ask "what would make this 10x better for 2x the effort?" The answer to "should we also build X?" is "yes, if it serves the vision." You have permission to dream.
* HOLD SCOPE: You are a rigorous reviewer. The plan's scope is accepted. Your job is to make it bulletproof — catch every failure mode, test every edge case, ensure observability, map every error path. Do not silently reduce OR expand.
* SCOPE REDUCTION: You are a surgeon. Find the minimum viable version that achieves the core outcome. Cut everything else. Be ruthless.
Critical rule: Once the user selects a mode, COMMIT to it. Do not silently drift toward a different mode. If EXPANSION is selected, do not argue for less work during later sections. If REDUCTION is selected, do not sneak scope back in. Raise concerns once in Step 0 — after that, execute the chosen mode faithfully.
Do NOT make any code changes. Do NOT start implementation. Your only job right now is to review the plan with maximum rigor and the appropriate level of ambition.

## Prime Directives
1. Zero silent failures. Every failure mode must be visible — to the system, to the team, to the user. If a failure can happen silently, that is a critical defect in the plan.
2. Every error has a name. Don't say "handle errors." Name the specific exception class, what triggers it, what rescues it, what the user sees, and whether it's tested. rescue StandardError is a code smell — call it out.
3. Data flows have shadow paths. Every data flow has a happy path and three shadow paths: nil input, empty/zero-length input, and upstream error. Trace all four for every new flow.
4. Interactions have edge cases. Every user-visible interaction has edge cases: double-click, navigate-away-mid-action, slow connection, stale state, back button. Map them.
5. Observability is scope, not afterthought. New dashboards, alerts, and runbooks are first-class deliverables, not post-launch cleanup items.
6. Diagrams are mandatory. No non-trivial flow goes undiagrammed. ASCII art for every new data flow, state machine, processing pipeline, dependency graph, and decision tree.
7. Everything deferred must be written down. Vague intentions are lies. TODOS.md or it doesn't exist.
8. Optimize for the 6-month future, not just today. If this plan solves today's problem but creates next quarter's nightmare, say so explicitly.
9. You have permission to say "scrap it and do this instead." If there's a fundamentally better approach, table it. I'd rather hear it now.

## Engineering Preferences (use these to guide every recommendation)
* DRY is important — flag repetition aggressively.
* Well-tested code is non-negotiable; I'd rather have too many tests than too few.
* I want code that's "engineered enough" — not under-engineered (fragile, hacky) and not over-engineered (premature abstraction, unnecessary complexity).
* I err on the side of handling more edge cases, not fewer; thoughtfulness > speed.
* Bias toward explicit over clever.
* Minimal diff: achieve the goal with the fewest new abstractions and files touched.
* Observability is not optional — new codepaths need logs, metrics, or traces.
* Security is not optional — new codepaths need threat modeling.
* Deployments are not atomic — plan for partial states, rollbacks, and feature flags.
* ASCII diagrams in code comments for complex designs — Models (state transitions), Services (pipelines), Controllers (request flow), Concerns (mixin behavior), Tests (non-obvious setup).
* Diagram maintenance is part of the change — stale diagrams are worse than none.

### Section 2: Error & Rescue Map
This is the section that catches silent failures. It is not optional.
For every new method, service, or codepath that can fail, fill in this table:
```
  METHOD/CODEPATH          | WHAT CAN GO WRONG           | EXCEPTION CLASS
  -------------------------|-----------------------------|-----------------

                           | API returns malformed JSON  | JSON::ParserError

  -----------------------------|-----------|------------------------|------------------
  Faraday::TimeoutError        | Y         | Retry 2x, then raise   | "Service temporarily unavailable"
  RateLimitError               | Y         | Backoff + retry         | Nothing (transparent)
  JSON::ParserError            | N ← GAP   | —                      | 500 error ← BAD
  ConnectionTimeoutError       | N ← GAP   | —                      | 500 error ← BAD
```

Rules for this section:
* `rescue StandardError` is ALWAYS a smell. Name the specific exceptions.
* `rescue => e` with only `Rails.logger.error(e.message)` is insufficient. Log the full context: what was being attempted, with what arguments, for what user/request.
* Every rescued error must either: retry with backoff, degrade gracefully with a user-visible message, or re-raise with added context. "Swallow and continue" is almost never acceptable.
* For each GAP (unrescued error that should be rescued): specify the rescue action and what the user should see.
* For LLM/AI service calls specifically: what happens when the response is malformed? When it's empty? When it hallucinates invalid JSON? When the model returns a refusal? Each of these is a distinct failure mode.

### Key Architecture Decision: REST vs. WebSocket RPC

The CRM already has RPC methods via `LiveCrmService`. Two options:

**Option A: New REST endpoints `/api/crm/*`** (RECOMMENDED)
- Follows CLAUDE.md convention: "Each UI tab gets its own API namespace"
- Simpler client-side code (fetch + JSON)
- Cacheable, bookmarkable, standard HTTP semantics
- Consistent with how every other page (sessions, images, sandbox, config) works

**Option B: Use existing WebSocket RPC methods**
- Already implemented (contact_create, contact_list, etc.)
- Requires WebSocket connection to be established first
- Harder to handle loading states, errors, retries
- Not how any other page tab works

**Decision needed from reviewer:** REST is the clear winner per codebase conventions, but it means ~15 new handlers in `api.rs` or a new `crm_routes.rs` module. This is the right investment.

```
  ------------------------------|--------------------------------|------------------

                                | Stale update (concurrent)      | Race condition
  DELETE /api/crm/contacts/:id  | Contact has matters            | FK constraint
  GET /api/crm/matters          | Invalid filter params          | Deserialization
  POST /api/crm/matters         | Referenced contact not found   | FK violation
  POST /api/crm/interactions    | Referenced contact not found   | FK violation
                                | Referenced matter not found     | FK violation
  ────────────────────────────────────────────────────────────────────────

  ERROR TYPE                     | RESCUED? | RESCUE ACTION            | USER SEES
  -------------------------------|----------|--------------------------|------------------
  sqlx::Error (pool)             | Y        | Return 503 + retry hint  | "Service busy"
  sqlx::Error (constraint)       | Y        | Return 409 + context     | "Contact has matters"
  Deserialization                | Y        | Return 400 + field name  | "Invalid field: stage"
  None result (not found)        | Y        | Return 404               | "Contact not found"
  Race condition (stale update)  | N ← GAP  | Need optimistic locking  | Silent data overwrite
```

**CRITICAL GAP:** No optimistic locking on updates. If two browser tabs edit the same contact, last-write-wins silently.

**Recommendation:** Add `updated_at` to update requests. Compare before writing. Return 409 if stale. This is a **HOLD for now** — document as a known limitation and add TODO for optimistic locking.

### Feature Gate Decision

CRM is gated behind `#[cfg(feature = "crm")]` in gateway. The web UI needs to handle:
- **CRM enabled:** Show the CRM tab and nav link
- **CRM disabled:** Hide the tab entirely (no dead nav link)

**Approach:** Add a `crm_enabled: bool` field to `GonData`. JS checks this before rendering the nav link and registering the route. This matches how `graphql_enabled` works.

### Production Failure Scenarios

| Integration Point | Failure Scenario | Handled? |
|-------------------|-----------------|----------|
| `sendRpc("crm.contacts.list")` | WebSocket not connected | Yes — `sendRpc` returns `{ok: false}` when WS is down. Page should show connection error. |
| `sendRpc("crm.contacts.upsert")` | Validation error (empty name) | Yes — `LiveCrmService` returns `ServiceError::message(...)`. Page should show toast. |
| `sendRpc("crm.contacts.upsert")` | Duplicate external_id | Yes — SQLite constraint error → ServiceError. |
| `sendRpc("crm.contacts.delete")` | Contact has matters with FK | **Partial** — `ON DELETE SET NULL` in migration means matters become orphaned, not rejected. This is by design (matters survive contact deletion). |
| NavCounts CRM query | CRM feature disabled | **Need check** — if CRM is disabled, `gw.services.crm` is `NoopCrmService` which returns empty results. NavCounts will show 0. Correct behavior. |

### Security

- **Auth:** All RPC methods go through the auth-gated WebSocket. No new auth surface.
- **PII:** `LiveCrmService.contact_to_json()` already calls `expose_secret()` on email/phone for JSON serialization. This is correct — only authenticated users reach this code.
- **XSS:** Preact auto-escapes all rendered content. Do NOT use `dangerouslySetInnerHTML` or `innerHTML` anywhere in page-crm.js.
- **LIKE injection in search:** `SqliteCrmStore.list_filtered` uses parameterized queries with `%term%` pattern. The `%` and `_` wildcards in user input are NOT escaped — searching for `%` would match everything. **Low severity** — single-tenant app, authenticated users only.

## Section 4: Performance Review

### N+1 Queries
- **Contact list:** `list_filtered()` returns flat contact rows — no N+1. Good.
- **Contact detail:** Needs separate calls for channels, matters, interactions. Use `Promise.all()` to parallelize, not sequential awaits.

### Search Debounce
- Must debounce search input at 250-300ms. The projects page uses 200ms for path completion — 300ms is appropriate for search (more typing expected).

### Pagination
- `list_filtered()` supports offset/limit. Default to 25 contacts per page.
- **Do NOT load all contacts then filter client-side.** Always use server-side filtering.

### Memory
- Contact list with 25 items per page: negligible.
- Contact detail with channels/matters/interactions: all loaded on demand, not cached globally.
- No concern.

### NavCounts
- The `build_nav_counts()` function runs concurrent `tokio::join!`. Adding a CRM count query adds ~1ms latency. No concern.
- **Implementation:** Add `gw.services.crm.list_contacts()` call to the join block. This returns ALL contacts — but for a count, we only need the array length. If the CRM has 1000+ contacts this becomes wasteful. **Alternative:** Add a `count_contacts()` method to CrmService that returns just the count via `SELECT COUNT(*)`. **Decision:** Use `list_contacts().len()` for v1 (matches how other counts work — providers, channels, etc. all load full lists). Add count-specific method later if perf becomes an issue.

### Key Architecture Decision: REST vs. WebSocket RPC

The CRM already has RPC methods via `LiveCrmService`. Two options:

**Option A: New REST endpoints `/api/crm/*`** (RECOMMENDED)
- Follows CLAUDE.md convention: "Each UI tab gets its own API namespace"
- Simpler client-side code (fetch + JSON)
- Cacheable, bookmarkable, standard HTTP semantics
- Consistent with how every other page (sessions, images, sandbox, config) works

**Option B: Use existing WebSocket RPC methods**
- Already implemented (contact_create, contact_list, etc.)
- Requires WebSocket connection to be established first
- Harder to handle loading states, errors, retries
- Not how any other page tab works

**Decision needed from reviewer:** REST is the clear winner per codebase conventions, but it means ~15 new handlers in `api.rs` or a new `crm_routes.rs` module. This is the right investment.

### Coupling
- New coupling: `moltis-web` → `moltis-crm` (through `GatewayState.services.crm`)
- This coupling already exists implicitly since the gateway owns the CRM store

```
  -------------------------------|----------|--------------------------|------------------
```

**CRITICAL GAP:** No optimistic locking on updates. If two browser tabs edit the same contact, last-write-wins silently.

**Recommendation:** Add `updated_at` to update requests. Compare before writing. Return 409 if stale. This is a **HOLD for now** — document as a known limitation and add TODO for optimistic locking.

## Section 3: Security & Threat Model

| Threat | Likelihood | Impact | Mitigated? |
|--------|-----------|--------|------------|
| PII exposure in API responses (email/phone) | Med | High | PARTIAL — `Secret<String>` prevents log leaks, but API must serialize to JSON for display. Need deliberate exposure. |

| XSS via contact name/metadata display | Med | Med | Mitigated by Preact (auto-escapes). Must NOT use `innerHTML`. |
| CSRF on mutation endpoints | Low | Med | Mitigated by existing auth middleware (`RequireAuth`). |
| SQL injection via search parameter | Low | High | Mitigated by sqlx parameterized queries (already in SqliteCrmStore). |
| Mass deletion via API | Low | High | No confirmation step in API. UI should confirm, but API is unprotected. |

**Key decision:** PII fields (email, phone) are `Secret<String>` in the backend. The API endpoints need to deliberately call `expose_secret()` to include them in JSON responses. This is correct — the authenticated user viewing their own CRM should see the data. But:
- **Never log** the response body for CRM endpoints
- **Never cache** CRM API responses in service workers
- Consider masking in the UI with reveal-on-click (delight opportunity)
